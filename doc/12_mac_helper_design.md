# 面向普通用户的 Mac Helper 设计

## 1. 文档目的

本文件定义 PDFreader 在 `P3+` 阶段的 `Mac helper companion` 方案，解决以下问题：

- 如何把“局域网 URL 书源”从开发者测试能力升级成普通用户可用能力
- 如何让用户只共享自己选中的文件夹，而不是整台 Mac
- 如何保留当前 `remote_url + library.json` 协议，不把主工程绑死在临时脚本上
- 如何在 `PWA` 约束下，明确 helper 的真实边界与分阶段落地方式

本稿的重点不是“现在就实现”，而是先把产品判断、架构边界和阶段目标钉住。

## 2. 当前问题

当前 `remote_url` 协议已经可用，但普通用户路径仍然不成立。

现状：

- 当前远程书源依赖一个可通过 `URL` 访问的静态目录
- `library.json`、`CORS`、端口和目录结构仍需要人工理解
- 开发测试可通过临时静态服务解决
- 这不适合作为正式用户方案

更关键的是，用户真正想要的是：

- 在 Mac 上选一个或几个文件夹
- 只共享这些文件夹里的 PDF
- 不碰终端
- 不手写 `library.json`
- 不理解 `CORS / HTTPS / 端口`

此外，macOS 系统自带的“文件共享”虽然可以只共享指定文件夹，但它本质是 `SMB` 文件共享，不是当前浏览器书源直接消费的 `HTTP/HTTPS + library.json` 协议，因此不能直接替代当前 `remote_url` 实现。

## 3. 产品判断

### 3.1 保留 remote_url，新增 helper 作为普通用户入口

产品不应放弃当前 `remote_url` 协议。

原因：

- 它已经和当前书库模型、同步逻辑、阅读打开路径对齐
- 它仍然是最容易跨平台扩展的远程来源抽象
- 后续无论兼容 `OPDS`、原生 App、桌面 helper，底层都仍可复用 URL 书源模型

因此应做的不是“换协议”，而是：

- 保留 `remote_url` 作为底层来源协议
- 新增一个 `Mac helper`，负责为普通用户生产并托管一个兼容的来源

### 3.2 手工静态服务降级为开发 / 高级用户路径

现有“手工准备目录 + 启静态服务”的方法仍保留，但定位必须降级为：

- 开发验证
- 协议联调
- 高级用户临时自建

它不再被视为正式用户入口。

### 3.3 helper 必须只共享用户显式选择的目录

这是普通用户信任感的前提。

helper 的共享边界必须做到：

- 默认不共享任何内容
- 只有用户手动添加的目录才会被扫描与暴露
- 目录外文件不可访问
- 只读共享，不回写原文件
- 支持随时停止共享

## 4. 设计目标

helper 方案要满足以下目标：

- `3 分钟内完成首次配置`
- `不需要终端`
- `不需要手写 library.json`
- `不需要理解 CORS / 端口 / 静态服务`
- `只共享选中的文件夹`
- `新增或删除 PDF 后可自动更新`
- `在手机或平板上能清楚知道该连接哪个地址`
- `出错时有明确提示，而不是浏览器报错`

## 5. 明确不做

以下能力不进入 helper 第一阶段：

- Windows / Linux / NAS 通用版
- 远程互联网访问
- 登录账号与权限系统
- 把阅读进度或元数据回写到 Mac 文件系统
- 修改 PDF、重命名源文件、移动源文件
- 自动发现全网设备
- 一开始就兼容 `OPDS`

第一阶段只解决：

- `Mac -> 同一局域网内的 PDFreader 客户端`
- `选目录 -> 开始共享 -> 读取 PDF`

## 6. 用户与使用场景

### 6.1 主用户

- 家里有一台 Mac 存书
- 想在 iPhone / iPad / 同一台 Mac 上阅读
- 愿意装一个小 helper，但不愿意折腾脚本

### 6.2 核心场景

场景 A：

- 用户在 Mac 上打开 helper
- 选择 `~/Documents/Comics`
- 点击“开始共享”
- 手机上打开 PDFreader 并连接该书源

场景 B：

- 用户后来新增几本 PDF 到共享目录
- helper 自动重新扫描
- 手机端再次同步即可看到新增内容

场景 C：

- 用户不想共享时点击“停止共享”
- 局域网地址立即失效
- PDFreader 中的书籍记录保留，但状态变为不可用

## 7. 体验方案

## 7.1 helper 首页只做一件事

helper 的首页必须极简。

推荐只保留以下信息：

- 当前共享状态：未共享 / 扫描中 / 已共享 / 出错
- 已选择的文件夹列表
- 主按钮：`添加文件夹`
- 主按钮：`开始共享 / 停止共享`
- 当前局域网地址
- 二维码
- 最近扫描结果：共多少本 PDF，上次更新时间

不应在第一版加入复杂的书库管理界面。

helper 不是另一个阅读器，而是来源提供器。

## 7.2 用户操作流

首次使用建议压缩为 4 步：

1. 安装并打开 helper
2. 选择一个或多个文件夹
3. 点击“开始共享”
4. 在另一台设备打开提供的地址或扫描二维码

后续日常使用应退化为：

- helper 常驻后台
- 文件变化后自动刷新
- 客户端点击“立即同步”

## 7.3 共享范围表达

为了降低用户的不安全感，UI 中必须明确显示：

- 当前只共享以下文件夹
- 不会共享整个磁盘
- 只读，不会修改原文件

如果用户移除某个文件夹，应明确提示：

- 该目录下的书将从远程目录中消失
- 已导入客户端的记录会保留，但会变成不可用

## 8. 关键产品边界

## 8.1 普通用户入口不能暴露 library.json 概念

`library.json` 仍然存在，但应完全隐藏在 helper 内部。

用户不应被要求：

- 手动创建 JSON
- 记字段名
- 自己算 `contentHash`

这些都应由 helper 自动完成。

## 8.2 “系统文件共享”不是正式方案

即使 macOS 自带文件共享足够简单，也不应把它作为当前产品正式入口。

原因：

- 它共享的是文件系统，不是当前浏览器直接消费的书源协议
- 它无法自动生成当前项目需要的 `library.json`
- 它无法直接承接封面、元数据和同步体验

因此 helper 应走自己的来源提供路径，而不是指导用户去系统设置里开 SMB。

## 8.3 PWA 下不能把 HTTP 局域网资源当默认能力

当前主产品仍是 `HTTPS` 部署的 `PWA`。

这意味着：

- helper 如果只暴露一个 `HTTP library.json`，并不自动等于“普通用户可用”
- 对 `iPhone / iPad` 的正式支持，不能建立在“用户自己理解浏览器安全策略”的前提上

因此 helper 的消费者方案必须分阶段：

- `阶段 A`：把来源生产、目录选择、自动扫描先做对
- `阶段 B`：把普通用户的实际连接路径做简单
- `阶段 C`：再解决安装态与更稳的移动端体验

## 9. 架构决策

## 9.1 helper 不是“只生成文件”，而是“生成 + 托管”

helper 不应只负责生成 `library.json`，还应负责托管。

推荐结构：

- 本地索引器：扫描用户选中的目录
- 元数据提取器：读取 PDF 基础信息并生成封面
- manifest 生成器：输出兼容当前协议的 `library.json`
- 本地 HTTP 服务：只读暴露 `library.json`、PDF 和封面
- helper UI：显示状态、地址、二维码、错误信息

这样用户不需要自己再找一个静态服务器。

## 9.2 helper 需要维护自己的本地索引数据库

如果 helper 每次只按文件路径现算，目录重命名或移动时很容易让同一本书被识别成新条目。

因此 helper 应维护一份轻量本地状态，至少记录：

- 已发现文件
- 上次扫描时间
- 文件尺寸与修改时间
- 已计算的 `contentHash`
- helper 内部分配的稳定书籍 ID
- 生成的封面缓存路径

这样可以做到：

- 扫描提速
- 避免每次都重新算 hash
- 文件名变化后尽量维持同一条目身份

## 9.3 第一阶段仍兼容当前 remote_url 协议

helper 产出的远程目录应与当前实现完全兼容：

- 根目录有 `library.json`
- 每本书返回 `id`
- 每本书返回 `contentHash`
- `pdfPath` 指向可直接读取的 PDF
- `coverPath` 为可选

这样客户端几乎不需要改协议层。

## 9.4 helper 默认提供两种入口，但只突出一种

推荐区分：

- `Consumer Mode`
  - helper 直接展示可连接地址和二维码
  - 面向普通用户
- `Advanced Source Mode`
  - 暴露底层来源地址
  - 仅用于调试或高级用户

在 UI 上只突出 `Consumer Mode`。

## 10. 连接方式设计

## 10.1 推荐默认走“helper 提供地址”

第一阶段最现实的默认方式是：

- helper 启动后显示一个局域网地址
- 用户在手机 / 平板上直接打开这个地址

这比要求用户先理解“书源地址”和“App 地址”更简单。

## 10.2 同源优先，降低跨域与配置成本

普通用户模式下，helper 最好不仅提供书源目录，还提供一个最小接入页，避免用户先理解“哪一个 URL 才是要填进 PDFreader 的 baseUrl”。

建议 helper 至少提供：

- 一个 `欢迎页 / 连接页`
- 一个可复制的来源地址
- 一个二维码
- 如果现网 `PWA` 与 helper 来源在移动 Safari 上联调不稳，则 H1 直接托管一份最小 PDFreader Web 壳，保证界面与来源同源

这里的产品判断是：

- 第一阶段先把“连接哪个地址”说清楚
- 优先保证普通用户能连接成功，而不是维持“所有流量都必须先走公网 App 壳”的洁癖

## 10.3 移动端安装态不是 helper 第一阶段承诺

如果仍以公网 `HTTPS PWA` 作为主 App 壳，那么移动端的最终安装态体验还会受浏览器安全边界影响。

因此 helper 第一阶段面向普通用户的承诺应收敛为：

- `Mac 浏览器可用`
- `iPhone / iPad Safari 可用`
- `局域网内直接访问流程可理解`

而不是提前承诺：

- `主屏安装态一定零配置可用`

这部分应在后续两个方向中二选一继续收口：

- helper 增加更完整的本地安全接入方案
- `P4` 进入原生壳后，由原生能力承接局域网来源

## 11. helper 内部模块

## 11.1 Folder Registry

职责：

- 保存用户选择的共享目录
- 校验目录是否仍存在
- 控制是否参与当前共享

## 11.2 Library Scanner

职责：

- 递归发现 PDF
- 过滤非 PDF
- 检测新增、删除、修改

建议只扫描用户勾选目录，不扫描整个 Home 目录。

## 11.3 PDF Inspector

职责：

- 提取标题
- 计算页数
- 读取首页尺寸
- 生成封面
- 计算 `contentHash`

这里建议后续从当前前端工程里抽一份“纯数据层能力”，避免 helper 与主工程各自维护两套 PDF 元数据逻辑。

## 11.4 Manifest Builder

职责：

- 为每本书生成稳定 `id`
- 输出 `library.json`
- 维护更新时间
- 生成 `pdfPath` / `coverPath`

## 11.5 Local Server

职责：

- 在局域网内提供只读访问
- 默认只绑定本机和局域网网卡
- 不做公网穿透
- 暴露健康状态接口供 UI 使用

## 11.6 Share UI

职责：

- 选择目录
- 显示共享状态
- 呈现错误信息
- 显示地址与二维码
- 手动重新扫描

## 12. 数据与协议策略

## 12.1 书籍 ID 不能只用文件路径

helper 生成的 `books[].id` 不能简单等于相对路径。

原因：

- 用户重命名文件后会直接变成新条目
- 目录移动后也会导致来源键变化

建议策略：

- helper 内部维护稳定 ID 映射
- `contentHash` 作为内容身份
- `books[].id` 作为来源实例内的稳定来源键

## 12.2 删除语义

当文件从共享目录消失时：

- helper 在 manifest 中移除该书
- 客户端同步后把原记录标记为不可用
- 不主动删除客户端本地元数据和进度

这与当前书库模型保持一致。

## 12.3 封面策略

helper 应优先本地生成封面并缓存。

原因：

- 客户端不必重复为每本远程书做封面首屏渲染
- 书库首屏可更快出现视觉结果

若封面生成失败：

- 不阻塞整本书进入目录
- 仅把 `coverPath` 置空

## 13. 安全与隐私

helper 第一阶段的安全原则：

- 默认关闭共享
- 只有显式添加的目录可读
- 不暴露目录外路径
- 不支持写入
- 不做公网映射
- UI 清晰显示“当前正在共享”

后续可以再补：

- 局域网访问口令
- 一次性配对码
- 指定网段访问

但不作为第一阶段前置条件。

## 14. 建议技术路线

从当前工程现实出发，helper 最务实的方向是：

- 单独做一个 `Mac companion app`
- UI 仍可使用 `React + TypeScript`
- 本地能力由桌面容器承接

推荐顺序：

1. 先用最快路径做出 `Mac helper alpha`
2. 跑通目录选择、扫描、托管和局域网访问
3. 再决定是否继续保持该技术栈，或切到更原生的实现

在当前阶段，不建议先为 helper 追求“最优技术选型”而拖慢验证。

## 15. 分阶段落地

## 15.1 H1：helper alpha

目标：

- 先让普通用户不碰终端也能共享目录

范围：

- 选择文件夹
- 手动开始 / 停止共享
- 生成 `library.json`
- 本地托管 PDF 与封面
- 显示地址与二维码
- helper 提供统一接入页
- 如有必要，托管一份最小 PDFreader Web 壳

验收标准：

- 完全不依赖项目脚本
- 不要求用户手工编辑目录结构之外的任何配置
- 至少一台 iPhone / iPad Safari 能通过 helper 提供的统一入口完成连接与同步

## 15.2 H2：helper beta

目标：

- 把普通用户路径进一步做顺

范围：

- 自动监听文件变化
- 更明确的错误说明
- helper 内欢迎页 / 连接页
- 一键复制来源地址
- 开机后恢复上次共享状态

验收标准：

- 用户不需要再次理解“library.json 是什么”
- 目录变动后能自动反映到客户端同步结果中

## 15.3 H3：consumer release

目标：

- 收口移动端安装态与长期稳定性

候选方向：

- helper 补更完整的本地安全接入
- 配合 `P4` 原生壳承接局域网来源

在做出方向判断前，不承诺“主屏安装态零配置”。

## 16. 与当前工程的衔接

helper 方案对当前主工程的影响应尽量收敛：

- `remote_url` 适配器继续保留
- `library.json` 协议继续保留
- 客户端只需增加更贴近 helper 的引导文案
- 主工程不应依赖 helper 才能运行

也就是说：

- helper 是普通用户入口增强
- 不是主工程协议重写

## 17. 当前结论

面向普通用户的正确方向不是继续教用户“怎么起静态服务”，而是：

- 保留当前 `remote_url` 协议
- 新增一个只共享所选目录的 `Mac helper`
- 把 `library.json`、封面生成、CORS、目录托管全部藏到 helper 内部

同时必须接受一个现实：

- helper 能先解决“普通用户如何把 Mac 目录变成书源”
- 但不应在第一版就夸口“移动端安装态零配置完全解决”

先把普通用户的来源创建路径做对，再收口移动端安装态，是更稳的路线。
