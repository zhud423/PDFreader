# PDFreader 产品路线图

## 1. 项目定位

PDFreader 是一个面向 `iPhone / iPad Safari + 主屏幕安装` 的漫画阅读产品，首期聚焦 `PDF 格式长竖页漫画`。

产品目标不是做全能书库，而是先把以下主链路做稳：

`打开应用 -> 继续上次阅读 -> 顺滑阅读 -> 退出后保留进度`

## 2. 核心原则

### 2.1 产品原则

- 阅读优先：允许打开大 PDF 慢一点，但阅读中必须顺滑
- 本地优先：默认不依赖后端；书库元数据、封面缓存、进度保存在本地
- 轻资产：不保存 PDF 副本，避免占用 iPad / iPhone 空间
- 渐进增强：先 PWA，后 App 化，不在 P1-P3 提前背原生成本

### 2.2 明确不做

以下能力至少在 P1-P3 不进入主范围：

- 搜索书内内容
- 密码 PDF
- 批注
- 自动裁边
- 自动滚动
- 多设备同步

## 3. 关键产品判断

### 3.1 文件来源

产品最终会支持两类来源：

- 本地手动上传 PDF
- 局域网 URL 书源

其中：

- `P1` 以本地手动上传为主链路
- `P2` 开始接入局域网 URL 书源
- `P3` 再把书库与来源管理做完整
- 面向普通用户的 Mac 局域网书源入口，应由 helper companion 承接，而不是要求手工搭静态服务

### 3.2 进度模型

产品需要同时记住：

- 上次打开的书
- 每本书的精确阅读位置

后续再逐步加入：

- 缩放状态
- 在读 / 已读状态
- 进度条与状态展示

### 3.3 部署与演进

- `P1-P3`：PWA，推荐部署到 `Cloudflare Pages`
- `P4`：在保留同一套前端核心代码的前提下，用 `Capacitor` 进入 App 化
- `P5`：视市场反馈决定是否增加账号、同步、商业化配套

## 4. 五期规划

## P1：自用 MVP

### 目标

验证最小闭环是否成立，重点确认：

- iOS / iPadOS Safari PWA 安装链路是否顺畅
- 长竖页 PDF 是否能稳定顺滑阅读
- 进度恢复模型是否可用

### 范围

- PWA 基础壳
- 首页继续阅读
- 本地手动上传 PDF
- 自动截取 PDF 首页作为封面
- 长竖页手动上下滑动阅读
- 基础缩放
- 横竖屏适配
- 单击隐藏 / 显示工具栏
- 自动隐藏状态操作区
- 默认沉浸式阅读体验
- 记住上次打开的书
- 记住每本书的精确滚动位置
- 缺失文件保留条目并显示不可用

### 阶段结果

用户可以在 iPhone / iPad 上把它当成日常阅读器开始使用，但书库和来源管理仍很轻。

## P2：公开测试版（80% 可用）

### 目标

把“自用工具”升级成“能给外部测试用户试用”的产品。

### 范围

- 接入局域网 URL 书源
- 本地来源与远程来源统一书籍模型
- 书源离线时保留条目并显示不可用
- 双击放大
- 记住缩放状态
- 长页分段渲染
- 阅读预加载与缓存策略
- 阅读器性能优化
- 打开大 PDF 的错误恢复与重试机制

### 阶段结果

产品达到约 80% 可用状态，适合面向小范围测试用户。

## P3：正式市场版

### 目标

补齐完整书库与来源管理，系统性修复 P1-P2 暴露的问题，达到正式发布标准。

### 范围

- 完整书库页
- 最近阅读
- 搜索
- 排序
- 重命名书库显示名
- 从书库移除但不删除原文件
- 缺失文件重新关联
- 收藏
- 在读 / 已读状态
- 进度展示
- 批量导入与去重
- 局域网书源稳定性收尾
- 面向普通用户的 Mac helper companion
- 安装引导、错误态和性能收尾

### 阶段结果

产品达到相对完善状态，可正式进入市场。

## P4：App 化与原生体验增强

### 目标

在保留已有产品内核的前提下，进入 App 形态，并补齐 PWA 难以承载的能力。

### 范围

- 使用 `Capacitor` 打包 iOS / Android App
- App 内打开本地文件
- 系统分享导入
- 原生文件权限与生命周期适配
- 深链与启动恢复
- 阅读配置档
- 隐私书架 / 应用锁
- 更自然的原生手势与反馈

### 阶段结果

产品不再局限于 PWA，进入 App 化阶段。

## P5：长期留存与生态能力

### 目标

从“阅读器”提升到“长期自用 / 可增长产品”。

### 范围

- 图像增强预设
- 系列 / 合集组织
- 阅读列表 / 待读队列
- 阅读统计
- 书库导出 / 导入
- 多书源统一管理
- 可选账号与多设备同步

### 阶段结果

产品具备长期留存和后续商业化扩展空间。

## 5. 阶段优先级说明

### 为什么 P2 先做远程书源和性能

用户明确不希望把大体积 PDF 副本存进 iPad / iPhone，本地上传只是 MVP 入口，长期主路径更接近“远程读取 + 本地保存元数据与进度”。因此，远程来源和阅读器性能比复杂书库更早进入主范围。

### 为什么 P3 再补完整书库

书库整理建立在来源模型稳定、阅读器体验稳定的基础上。过早做重书库，会把产品重心从“能顺滑读”偏成“会整理但不好读”。

### 为什么 P4 才开始 App 化

P1-P3 先验证产品价值与核心体验，P4 再承担原生化成本更合理；同时架构上从一开始就预留 App 化接口，避免重写。

## 6. 推荐技术方向

- 前端：`React + TypeScript + Vite`
- PWA：`vite-plugin-pwa`
- PDF 渲染：`pdf.js`
- 本地数据：`IndexedDB + Dexie`
- 部署：`Cloudflare Pages`
- App 化：`Capacitor`

## 7. 当前结论

当前项目以 `PWA 路线` 正式立项。后续详细设计默认围绕 `P1` 展开，并以 `P4 App 化可复用` 为前提约束。
