# P3 作品书库与章节阅读设计

## 1. 文档目的

本文件定义 `P3` 阶段的核心产品结构调整，统一以下三件事：

- 首页升级为 `作品书库首页`
- 上传与书源管理迁移到独立 `添加内容页`
- 阅读页菜单改为 `顶部 + 底部` 双层结构

同时，本稿修正当前产品语义：

- `作品` 才是用户理解中的“书”
- `PDF` 实际对应的是 `章节`
- 阅读进度应挂在 `章节` 上，并汇总到 `作品`

本稿只参考截图的视觉层次和操作密度，不抄截图里的平台功能。

## 2. 核心模型

## 2.1 一等实体

P3 起，界面和数据都应围绕两层实体组织：

- `TitleRecord`
  - 一部作品
  - 有自己的名称、封面、收藏状态、阅读状态
- `ChapterRecord`
  - 作品下的一章，对应一个 PDF 文件
  - 有自己的名称、封面、进度、来源状态

## 2.2 身份策略

章节身份继续使用现有 `canonicalKey` 思路，但语义调整为：

- 它是 `章节轻量身份键`
- 用于导入去重、章节重关联、多端后续兼容
- 不等同于强内容指纹

P3 当前规则：

- `canonicalKey` 不应包含 `file.name`
- 因为同一章节被复制、重命名后，不应被识别成不同章节

P3 先不切到真正的 `content hash`，避免提前增加复杂度。

## 2.3 本地优先边界

当前仍坚持：

- 文件保留在设备本地或远程来源中
- 应用只保存元数据、封面、进度和来源绑定
- 不引入中心化后端服务

因此 P3 做到的是：

- 单设备内的完整作品书库管理
- 为后续多端同步预留章节身份键

P3 不承诺：

- 无共享介质时的自动多端一致
- 把编辑后的名称和封面回写到远程文件系统

## 3. P3 设计目标

- 首页第一眼就是作品书库，而不是上传面板
- 用户按“作品”浏览，按“章节”阅读
- 上传、本地导入、局域网书源录入进入独立页面
- 阅读页菜单结构更接近漫画 App 心智
- 不引入社区、推荐、评论、弹幕、账号系统

## 4. 页面结构

P3 第一阶段建议保留 3 个主页面：

1. `作品书库首页 /`
2. `添加内容页 /add`
3. `章节阅读页 /reader/:chapterId`

动作层：

- 搜索层
- 排序 / 筛选面板
- 作品操作菜单
- 章节操作菜单
- 重命名弹窗
- 更换封面弹窗
- 从书库移除确认弹窗

P3 第一阶段仍不做独立作品详情页。
首页列表就是主承接页，避免增加跳层。

## 5. 首页：作品书库

## 5.1 页面目标

首页需要完成 6 个任务：

- 展示全部作品
- 提供继续阅读入口
- 展示作品级阅读状态
- 支持搜索、排序、筛选
- 承接作品管理动作
- 提供上传页入口

## 5.2 信息架构

从上到下建议固定为：

1. 页面标题与主动作
2. 一级视图切换
3. 二级筛选与排序
4. 作品列表
5. 悬浮添加入口

### 一级视图

保留两项：

- `书库`
  - 当前仍在管理的作品
- `历史`
  - 最近阅读过的作品

### 二级筛选

在 `书库` 下：

- 全部
- 收藏
- 在读
- 已读
- 不可用

在 `历史` 下：

- 最近打开
- 最近导入
- 最近同步

### 排序

- 最近阅读
- 最近更新
- 最近导入
- 作品名

## 5.3 列表项结构

首页展示的是 `作品卡片`，不是 PDF 卡片。

每个作品项最少包含：

- 作品封面
- 作品显示名
- 来源摘要：本地 / 局域网 / 混合
- 章节总数
- 最近阅读章节名
- 进度摘要：最近看到第几章、第几页
- 状态标签：可读 / 部分缺失 / 全部失效
- 主按钮：继续阅读 / 开始阅读 / 修复来源
- 次按钮：更多

不加入参考图中的热度、人气、更新榜、评论等平台字段。

## 5.5 首页交互说明

- 点击封面 / 标题：进入该作品的最近阅读章节
- 点击主按钮：执行最合适动作
- 点击 `•••`：打开作品菜单
- 点击 `添加`：进入 `/add`
- 点击 `搜索`：打开搜索层
- 点击 `排序`：打开底部排序面板

## 6. 章节列表层

虽然 P3 第一阶段不单做完整作品详情页，但作品菜单里需要一个轻量 `章节列表层`。

用途：

- 浏览某部作品下所有章节
- 对章节做重命名、改封面、删除记录、选择阅读

### 章节列表线框

```text
+--------------------------------------------------+
| < 返回                 作品名                    |

| 第 11 章 已读                            [打开]    |
| 第 10 章 已读                            [打开]    |
| 第 09 章 缺失                            [修复]    |
+--------------------------------------------------+
```

## 7. 添加内容页

## 7.1 页面目标

把当前首页上的工作台型能力全部迁出：

- 单本上传 PDF
- 批量导入 PDF
- 录入局域网书源
- 同步局域网书源

## 7.2 页面模块

建议分 4 块：

1. 添加到已有作品
2. 新建作品并上传第一章
3. 局域网书源
4. 最近操作结果

## 7.3 页面交互策略

本地导入时，用户需要明确选择：

- 导入到已有作品
- 或创建新作品

这是 `P3` 和 `P2` 最大的交互差异。
因为现在 PDF 是章节，不再能默认“一本 PDF 就是一部作品”。

```

## 7.5 书源导入规则

对局域网书源，P3 建议允许两种策略：

- `按远程目录自动建作品`
- `同步到已有作品`

第一阶段可以先只落地第一种，减少复杂度。

## 8. 阅读页：上下菜单结构

## 8.1 设计目标

参考截图的上下分层，但保持 PDFreader 自己的阅读逻辑：

- 菜单默认隐藏
- 中部点按切菜单
- 顶部负责上下文与导航
- 底部负责阅读控制
- 不引入评论、弹幕、互动能力

## 8.2 顶部菜单内容

- 返回
- 作品名
- 章节名
- 当前页 / 总页数
- 来源状态
- 更多菜单

## 8.3 底部菜单内容

- 进度滑杆
- 当前章节进度摘要
- 缩小
- 放大
- 回到上次进度点
- 阅读设置

阅读设置仅包含：

- 点按步进开关
- 步进距离
- 页面分隔细线开关

## 8.4 阅读页低保真线框

### 菜单隐藏态

|        下 1/3 点按：前进                         |

```

## 8.5 阅读页更多菜单

阅读页右上角更多菜单建议包含：

- 打开章节列表
- 修改章节名
- 修改章节封面
- 重试连接来源
- 从作品中移除本章

避免把重度作品管理动作塞进阅读页。

## 9. 页面流转

## 9.1 主链路

```text
书库首页
  -> 继续阅读
  -> 阅读页（某作品下某章节）
  -> 返回
  -> 书库首页
```

## 9.2 添加章节链路

```text
书库首页
  -> 添加内容页
  -> 选择已有作品 或 新建作品
  -> 导入章节 PDF
  -> 返回书库首页
```

## 9.3 修复来源链路

```text
书库首页 / 章节列表
  -> 不可用章节
  -> 重试连接 / 重新关联
  -> 恢复可用
  -> 继续阅读
```

## 10. P3 实现拆分建议

1. `模型与身份修正`
   - 明确作品 / 章节语义
   - `canonicalKey` 去掉文件名
   - 为作品级聚合视图预留字段

2. `信息架构调整`
   - 新增 `/add`
   - 首页从工作台改成作品书库首页
   - 上传能力迁出首页

3. `作品 / 章节管理`
   - 作品重命名、改封面
   - 章节重命名、改封面
   - 收藏、在读、已读
   - 从书库移除

4. `阅读页菜单改造`
   - 顶部导航条
   - 底部控制条
   - 章节列表层和阅读设置面板

## 11. 当前建议结论

P3 不应该继续沿用“一个 PDF 就是一整本书”的界面语义。
正确方向是：

- 首页展示作品
- PDF 作为章节被管理
- 上传页负责把章节导入到作品
- 阅读页展示作品名和章节名

这样更符合真实漫画使用场景，也和你刚补充的产品定义一致。
