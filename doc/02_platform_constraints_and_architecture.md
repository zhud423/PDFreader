# 平台约束与架构原则

## 1. 文档目的

本文件记录 PWA 路线下必须长期遵守的平台约束与架构边界，避免后续设计因为短期实现便利而走入死路。

## 2. 已确认技术路线

- `P1-P3`：PWA
- `P4`：通过 `Capacitor` 进入 App 化
- 当前推荐部署：`Cloudflare Pages`

结论：

前 3 期的实现必须以“未来能迁移到 App”为前提，而不是写成只能停留在浏览器里的临时方案。

## 3. 平台约束

## 3.1 iOS / iPadOS Safari 对本地文件访问的约束

浏览器环境下，本地文件访问能力并不等价于原生 App 的文件访问能力。

对本项目的直接影响：

- 本地上传适合作为导入入口
- 不适合作为长期稳定的一键继续阅读能力
- “不存副本”前提下，条目与进度可以长期保存，但源文件重新打开能力可能失效

因此产品设计必须内建：

- 不可用状态
- 重新选择文件
- 来源抽象层

## 3.2 HTTPS 与远程书源

PWA 以 HTTPS 站点提供应用壳。后续接入局域网 URL 书源时，应优先假设远程资源也通过 HTTPS 提供。

设计结论：

- 不把“HTTP 明文局域网资源”当默认能力
- 远程书源优先走可标准化的 URL 接入方式
- 后续优先评估 `OPDS` 兼容模型，而不是私有目录协议

## 3.3 不保存 PDF 副本

这是明确产品约束，不是临时实现选择。

直接影响：

- 本地存储只放元数据、进度、封面和缓存信息
- 大文件长期可用性更依赖远程来源或未来 App 化后的原生文件访问

## 4. 架构原则

## 4.1 Source Adapter

所有书籍来源必须通过统一的来源适配层接入。

建议的来源类型：

- `LocalUploadSource`
- `RemoteUrlSource`
- `UnavailableSource`
- `NativeFileSource`（P4 预留）
- `ShareImportSource`（P4 预留）

任何业务代码都不应直接依赖浏览器文件 API 或具体 URL 结构。

## 4.2 Reader Engine

阅读器内核只负责：

- 打开文档
- 渲染页面 / 分段
- 缩放与滚动
- 进度恢复

阅读器不负责：

- 文件选择
- 来源权限管理
- 书库业务逻辑

这样在 P4 App 化时，阅读器内核可以保持最大程度复用。

## 4.3 Library Repository

书库层统一管理：

- 书籍元数据
- 封面
- 可用状态
- 进度
- 最近阅读

当前实现可基于 `IndexedDB`，但接口设计上要允许未来替换为原生存储实现。

## 4.4 Platform Bridge

所有平台差异能力必须通过桥接层收口，例如：

- 文件选择
- PWA 安装提示
- 生命周期事件
- 未来的原生分享导入
- 深链
- 权限查询

不要把平台分支逻辑散落到页面组件中。

## 5. 从现在开始不能写死的点

### 5.1 不能把书籍唯一标识绑死在浏览器临时文件对象上

应综合来源类型、文件元数据、内容特征生成稳定标识，保证：

- 重复导入能识别
- 重新关联文件时能接回原进度
- P2 接入远程书源后仍能统一建模

### 5.2 不能把进度只记录成裸滚动值

建议至少保存：

- `bookId`
- `pageIndex`
- `segmentIndex`
- `scrollOffset`
- `zoomScale`
- `viewport`

否则后续切换设备尺寸、优化分段策略后会很难兼容。

### 5.3 不能让 Service Worker 承担业务数据职责

Service Worker 只负责：

- App Shell 缓存
- 静态资源缓存
- 基础离线入口

业务数据仍应由应用数据层管理。

### 5.4 不能把远程书源协议做成一次性私有实现

后续书源能力应优先考虑：

- 标准 URL 访问
- OPDS 兼容
- 可扩展到 Komga / Kavita 一类现成生态

## 6. App 化预留原则

## 6.1 P1-P3 需保证的可迁移性

- 业务状态与 UI 不依赖浏览器专属对象
- 文件来源通过适配器封装
- 本地数据访问通过 repository 封装
- 生命周期事件有统一入口

## 6.2 P4 App 化时希望直接复用的部分

- 书库 UI
- 首页逻辑
- 阅读器 UI
- PDF 渲染主逻辑
- 进度模型
- 封面生成逻辑

## 6.3 P4 预计需要新增的原生能力

- 系统分享导入
- App 内打开本地文件
- 权限处理
- 深链
- 更稳定的本地文件持久引用
- 原生缓存与生命周期优化

## 7. 当前设计结论

PWA 路线继续成立，但必须接受一个现实：

- `P1` 的本地上传是 MVP 入口，不是最终最强来源模型
- `P2` 的局域网 URL 书源是产品迈向 80% 可用的关键
- `P4` 的 App 化是文件接入和系统级体验真正补齐的阶段
