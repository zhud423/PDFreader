# 阅读器内核设计

## 1. 文档目的

本文件定义 PDFreader 在 `P1-P2` 阶段的阅读器内核设计，重点解决以下问题：

- 长竖页 PDF 如何稳定渲染
- 为什么不能按整页大图思路实现
- 如何设计分段渲染、预加载与缓存
- 如何保存和恢复阅读进度
- 如何为 `P4` App 化保留复用空间

## 2. 设计目标

- `阅读中顺滑优先`
- `大文件首开允许稍慢`
- `长竖页内存可控`
- `来源无关`
- `进度恢复稳定`
- `未来 App 化可复用`

## 3. 已知输入特征

结合样例文件和用户描述，当前需要优先支持的 PDF 特征是：

- 页数较少，通常 `< 10`
- 单页极高，典型为长条漫画页
- 文件体积大，常见为几十 MB 到几百 MB
- 主要阅读方式是纵向连续滚动

结论：

真正的难点不是“总页数很多”，而是“单页极高，单页渲染代价非常大”。

## 4. 为什么不能整页渲染

如果把一页超长 PDF 直接渲染成单个大 canvas 或单张位图，会遇到 4 类问题：

- 浏览器 canvas 尺寸限制
- 单次位图分配过大，容易触发内存压力
- 缩放时需要整页重渲染，交互卡顿
- 离屏页面仍占大量内存

因此，阅读器必须采用：

`页面级文档模型 + 分段级渲染模型`

## 5. 核心架构

阅读器内核建议分成以下模块：

- `DocumentLoader`
- `PageModelBuilder`
- `SegmentPlanner`
- `RenderScheduler`
- `SegmentCache`
- `ViewportController`
- `ProgressTracker`
- `ReaderStateStore`

## 6. 模块职责

## 6.1 DocumentLoader

职责：

- 加载来源数据
- 初始化 `pdf.js`
- 获取页数、页面尺寸、文档元数据

输入：

- `SourceAdapter` 提供的文档字节流、URL 或可读句柄

输出：

- `PdfDocumentHandle`
- 基础元数据

要求：

- 只做加载和解析准备
- 不直接参与 UI

## 6.2 PageModelBuilder

职责：

- 为每一页建立统一页面模型

页面模型建议至少包含：

- `pageIndex`
- `width`
- `height`
- `aspectRatio`
- `rotation`
- `estimatedSegmentCount`

说明：

- 即使当前多数文件页数很少，也不要把页面建模省掉
- 后续普通页 PDF、混合页 PDF 都需要统一入口

## 6.3 SegmentPlanner

职责：

- 根据页面高度、当前缩放和目标显示宽度，把单页拆成多个可独立渲染的 segment

输出建议：

- `segmentIndex`
- `pageIndex`
- `sourceRect`
- `targetHeight`
- `priority`

关键原则：

- segment 要足够小，便于缓存和释放
- segment 不能过小，否则会导致节点过多、滚动中频繁调度

P1 初始策略建议：

- 按固定视觉高度切段
- 以“当前视口高度的 1 到 2 倍”作为单段目标高度
- 后续在 P2 再按设备和缩放策略微调

## 6.4 RenderScheduler

职责：

- 决定什么时候渲染哪个 segment
- 管理渲染优先级
- 避免重复渲染

优先级建议：

1. 当前视口内 segment
2. 视口前后相邻 segment
3. 更远距离的预加载 segment

调度要求：

- 任何时刻都要能取消低优先级任务
- 缩放或快速滚动时，应立即废弃过时渲染任务

## 6.5 SegmentCache

职责：

- 缓存最近使用的 segment 位图 / canvas
- 在内存压力下优先释放远离视口的内容

缓存建议分两层：

- `hot cache`：当前和相邻 segment
- `warm cache`：短时间可能回看的 segment

释放策略建议：

- 根据距当前视口的距离淘汰
- 根据当前缩放等级淘汰不匹配缓存
- 在页面隐藏或切后台时主动收缩缓存

## 6.6 ViewportController

职责：

- 维护当前视口尺寸
- 监听滚动与缩放
- 通知渲染层需要的 segment 范围

注意：

- 不要让 React 高频 state 更新直接驱动滚动计算
- 视口计算和滚动事件处理应尽量靠近原生 DOM / observer 层

## 6.7 ProgressTracker

职责：

- 根据当前可见页面和 segment 计算阅读位置
- 节流写入进度
- 在切后台、返回、崩溃前尽量完成最后一次保存

P1 需要稳定保存：

- `bookId`
- `pageIndex`
- `segmentIndex`
- `scrollOffsetWithinSegment`
- `updatedAt`

P2 再强化：

- `zoomScale`
- `viewportSnapshot`
- `restoreStrategyVersion`

## 6.8 ReaderStateStore

职责：

- 持有 UI 层需要的阅读状态
- 隔离数据层和页面层

建议持有：

- 当前书籍信息
- 当前加载状态
- 当前渲染状态
- 当前错误状态
- 当前 UI 显隐状态

## 7. 来源抽象

阅读器内核不能直接依赖文件输入方式。

必须通过统一的 `SourceAdapter` 接口提供文档读取能力。

建议接口能力：

- `open()`
- `readMetadata()`
- `getDocumentInput()`
- `revalidate()`
- `getAvailabilityStatus()`

当前规划的来源类型：

- `LocalUploadSource`
- `RemoteUrlSource`
- `UnavailableSource`

未来 `P4` 追加：

- `NativeFileSource`
- `ShareImportSource`

## 8. 分段渲染策略

## 8.1 基础原则

- 不整页渲染
- 不一次性预渲染全部段
- 优先保证当前视区
- 远离当前视区的位图要及时释放

## 8.2 P1 策略

P1 以“稳”为目标，策略要保守：

- 首次进入时先渲染当前段
- 同步排队预渲染前后各 1 到 2 段
- 用户高速滚动时暂停远端预加载
- 缩放后允许短暂模糊或补渲染，但不能卡死

## 8.3 P2 策略

P2 以“顺滑”为目标，策略进一步强化：

- 基于滚动方向预测下一批 segment
- 加入更精细的缩放级别缓存
- 记录典型设备的最佳分段参数
- 加入长文件读取重试与降级策略

## 9. 进度保存与恢复

## 9.1 进度定义

P1 不建议把进度只记成一个全局滚动像素值，因为以下因素会改变绝对像素：

- 设备尺寸变化
- 方向变化
- 缩放变化
- 分段策略变化

因此，进度建议由 3 层组成：

- 页面层：当前在第几页
- 段层：当前在哪个 segment
- 偏移层：当前 segment 内偏移多少

## 9.2 保存时机

必须保存的时机：

- 滚动节流触发
- 页面进入后台
- 用户返回首页
- 阅读器异常退出前的可捕获时机

## 9.3 恢复策略

恢复顺序建议：

1. 找到对应页
2. 找到对应 segment
3. 等待该段可用
4. 恢复段内偏移
5. 如果段结构已变，按近似比例回退恢复

## 9.4 兼容策略

建议保存：

- `restoreStrategyVersion`

用途：

- 后续分段算法升级后，可以知道旧进度该如何迁移

## 10. 缩放设计

## 10.1 P1 范围

P1 只做基础双指缩放，不追求完整高级体验。

约束：

- 缩放应尽量不打断纵向阅读
- 不强求记住缩放
- 遇到重渲染时优先保证交互连续，不追求立即清晰

## 10.2 P2 范围

P2 补以下能力：

- 双击放大
- 记住缩放状态
- 缩放后保持更稳的滚动与显示

## 10.3 技术原则

- 缩放交互和高清重渲染分离
- 先响应手势，再异步补高质量图
- 缩放结束后再集中处理重渲染任务

## 11. 错误与降级

## 11.1 本地文件失效

处理方式：

- 终止读取
- 标记来源不可用
- 保留进度
- 引导重新选择文件

## 11.2 PDF 解析失败

处理方式：

- 告知当前文件无法解析
- 提供重试与返回
- 不写坏数据到书库记录

## 11.3 内存压力过大

处理方式：

- 清空 warm cache
- 收缩 hot cache
- 降低预加载段数
- 必要时降低渲染分辨率

## 12. 性能指标建议

P1 不必过度量化，但需要建立最低指标意识：

- 首次进入阅读页要有清晰加载反馈
- 当前视区 segment 渲染完成时间应稳定可接受
- 滚动过程中不能因为后台预加载造成明显掉帧
- 长时间阅读后内存占用不能持续只升不降

P2 开始再补设备实测基线。

## 13. 与 UI 层的边界

UI 层负责：

- 页面结构
- 加载态与错误态展示
- 工具栏显隐
- 交互触发

阅读器内核负责：

- 文档模型
- 分段渲染
- 缓存
- 进度计算
- 恢复逻辑

不要让页面组件直接决定分段和缓存策略。

## 14. 与 App 化的关系

本设计默认要在 `P4` 最大化复用，因此：

- 不把文档输入写死成浏览器 File
- 不把缓存写死在浏览器临时对象上
- 不把恢复逻辑写死在浏览器滚动实现上

未来进入 App 化后，理想状态是：

- 更换来源适配器
- 增加原生桥接
- 保留大部分阅读器逻辑不动

## 15. 分阶段实施建议

### P1 实现顺序

1. 接入 `pdf.js` 基础加载
2. 建立页面模型
3. 实现最小分段策略
4. 实现当前段与相邻段渲染
5. 接入滚动定位
6. 实现节流进度保存
7. 接入错误态与不可用态

### P2 优化顺序

1. 细化预加载策略
2. 加入缩放重渲染优化
3. 接入远程 URL 书源
4. 优化恢复策略
5. 建立性能实测与调参机制

## 16. 当前结论

对 PDFreader 而言，阅读器内核的正确方向不是“把 PDF 画出来”，而是：

`围绕长竖页建立一个可分段、可回收、可恢复的连续阅读引擎`

只要这层设计做对，P1 能顺利自用，P2 才有机会达到 80% 可用，P4 才可能平滑进入 App 化。
