# 发布与测试计划

## 1. 文档目的

本文件定义 PDFreader 在 `P1-P3` 阶段的测试策略、验收标准和发布节奏，目标是避免产品在“功能看起来有了”但体验和稳定性不达标时过早进入下一阶段。

## 2. 发布阶段定义

## 2.1 P1 自用 MVP

面向对象：

- 仅限自己使用

目标：

- 验证核心闭环
- 找出阅读器和本地上传的真实问题

## 2.2 P2 公开测试版

面向对象：

- 小范围测试用户

目标：

- 验证远程书源与阅读体验
- 判断是否达到 80% 可用

## 2.3 P3 正式市场版

面向对象：

- 正式用户

目标：

- 达到相对完整和稳定的市场可用标准

## 3. 测试维度

每个阶段都至少覆盖以下维度：

- 功能正确性
- 阅读体验
- 性能与内存
- 状态恢复
- 异常与容错
- 设备适配

## 4. 设备与环境矩阵

## 4.1 P1 最低测试矩阵

- `iPhone Safari`
- `iPad Safari`
- 已添加到主屏幕的 PWA 模式

说明：

- 浏览器内打开与主屏幕模式都要测
- 因为两者在生命周期和 UI 表现上可能不同

## 4.2 P2 扩展测试矩阵

在 P1 基础上增加：

- 局域网 Wi-Fi 环境
- 远程 URL 书源可达与不可达场景
- 不同大小 PDF
- 不同长宽比 PDF

## 4.3 P3 扩展测试矩阵

在 P2 基础上增加：

- 多书源
- 大量书籍记录
- 长时间连续阅读
- 长时间后台切换恢复

## 5. 测试样本策略

至少准备以下测试文件：

- `S1`：小体积、单页较短 PDF
- `S2`：中体积、典型长竖页 PDF
- `S3`：大体积、超长页 PDF
- `S4`：多页但页高差异明显的 PDF
- `S5`：故意损坏或异常 PDF

目的：

- 避免只对一种“理想文件”优化

## 6. P1 验收清单

## 6.1 功能验收

- 可以从设备选择至少一个 PDF
- 可以导入多个 PDF
- 导入后可生成封面
- 首页可以显示继续阅读
- 阅读页可以恢复到上次位置
- 条目不可用时可以正确展示状态
- 可以重新选择文件恢复书籍

## 6.2 体验验收

- 首页首屏明确聚焦继续阅读
- 阅读页进入后干扰少
- 单击切换工具栏逻辑稳定
- 横竖屏切换后不出现明显错位

## 6.3 性能验收

- 打开大 PDF 时有明确加载反馈
- 阅读中滚动无明显持续卡顿
- 长时间阅读后内存压力不会快速积累到不可用

## 6.4 容错验收

- 选错文件类型有明确提示
- PDF 解析失败不会导致应用崩坏
- 文件丢失后条目和进度仍保留

## 7. P2 验收清单

## 7.1 功能验收

- 可添加至少一个局域网 URL 书源
- 书源离线后条目保留但不可用
- 恢复在线后可继续阅读
- 双击放大可用
- 可记住缩放状态

## 7.2 体验验收

- 阅读页在长竖页场景下明显优于 P1
- 缩放后不会频繁出现严重错位
- 打开大文件虽慢，但进入后阅读连续性更好

## 7.3 80% 可用判断标准

如果以下条件同时成立，可认为达到 80% 可用：

- 主要阅读路径稳定
- 远程书源可连续使用
- 继续阅读可信
- 常见错误有承接
- 测试用户无需频繁手动修复数据

## 8. P3 验收清单

## 8.1 功能验收

- 书库搜索、排序、重命名、移除、重关联都可用
- 收藏、在读 / 已读、进度展示可用
- 多来源书籍模型保持一致

## 8.2 质量验收

- 首页、阅读页、书库页无明显断裂感
- 大部分异常路径都有明确文案和恢复动作
- 安装与首次使用引导足够清晰

## 8.3 市场发布判断标准

如果以下条件同时成立，可认为具备正式进入市场条件：

- 主要用户路径稳定
- 书库体验完整
- 远程书源稳定性可接受
- 自用场景中不再频繁遇到结构性问题
- 小范围测试反馈主要集中于优化，而不是基础不可用

## 9. 测试场景清单

## 9.1 首页场景

- 首次进入空状态
- 有继续阅读
- 无继续阅读但有书
- 最近阅读不可用
- 大量书籍列表滚动

## 9.2 导入场景

- 单本导入
- 多本导入
- 重复导入
- 错误文件导入
- 导入中断

## 9.3 阅读场景

- 初次打开
- 继续阅读恢复
- 长时间滚动
- 横竖屏切换
- 切后台再回来
- 工具栏显隐切换
- 缩放

## 9.4 异常场景

- 文件失效
- PDF 损坏
- 网络书源离线
- 渲染失败
- 数据恢复失败

## 10. 性能观察指标

P1-P3 建议持续记录以下指标，即使前期不做复杂监控，也至少要能手工观测：

- 文档首开耗时
- 首页进入阅读页耗时
- 当前视区 segment 渲染耗时
- 恢复到上次位置耗时
- 长时间阅读后缓存规模变化
- 失败率最高的操作类型

## 11. 发布节奏建议

## 11.1 P1

- 开发完成后先进行 3 到 5 天连续自用
- 期间只修阻塞阅读的问题
- 不在 P1 末期临时加功能

## 11.2 P2

- 先邀请少量固定测试用户
- 每次发布只聚焦一类核心变化，例如“远程书源”或“缩放优化”
- 收集问题时按严重级别分类，不混在需求池里
- 对外测试前，至少按 `10_p2_beta_runbook.md` 跑一遍手工回归

## 11.3 P3

- 发布前至少进行一轮完整回归
- 回归必须覆盖首页、阅读页、书库页、来源异常态
- 市场发布版本应冻结新增需求，优先收尾稳定性

## 12. 缺陷分级建议

- `P0`：无法导入、无法阅读、进度丢失、频繁崩溃
- `P1`：继续阅读失效、主要交互异常、书源普遍不可用
- `P2`：局部页面错误、状态文案错误、次要功能异常
- `P3`：视觉细节、边界表现优化

## 13. 当前建议结论

对这个项目来说，测试不能只看“功能是否存在”，而要重点看：

- 阅读是否真的顺
- 继续阅读是否可信
- 文件失效和书源失效时是否还能恢复

只要这三点没有稳定，产品就不应该轻易进入下一期。
