# P1 工程初始化与任务拆分

## 1. 当前脚手架结论

当前仓库已从纯文档状态进入可运行的 `P1` 工程基线阶段，已落地：

- `React + TypeScript + Vite` 基础工程
- `vite-plugin-pwa` 的基础离线壳与 manifest
- 首页与阅读页两条主路由
- `IndexedDB + Dexie` 数据层
- `LocalUploadSource` 来源适配器
- 本地 PDF 导入、封面生成、元数据入库
- 继续阅读入口与基础进度保存
- 本地文件失效后的“重新选择文件”恢复路径
- 长竖页 `segment` 分段渲染
- 首页安装提示、不可用条目区和多文件导入反馈
- `visibilitychange / pagehide` 生命周期保存桥接

当前状态已经接近 `P1` 的可自用 MVP，可进入真机连续自用和收尾阶段。

## 2. 当前目录结构

建议后续保持如下结构方向：

```text
src/
  app/          应用壳与路由
  components/   通用展示组件
  db/           Dexie 数据库定义
  domain/       领域模型
  lib/          PDF、会话级注册表等基础能力
  pages/        首页与阅读页
  services/     书库业务服务
  sources/      来源适配层
  shared/       通用工具函数
  styles/       全局样式
```

## 3. P1 收尾与验证任务

### T1 真机连续阅读验证

当前应重点验证而不是继续扩功能：

- iPhone Safari
- iPad Safari
- 主屏安装模式
- 长时间滚动与后台切换恢复

### T2 阅读性能参数收敛

当前已具备页面 + 段 + 偏移模型，但还应继续观察：

- segment 目标高度是否需要按设备调优
- 缩放后重布局的稳定性
- 长时间阅读后的内存波动

### T3 交互细节打磨

当前建议只做小修，不再扩功能：

- 工具栏显隐细节
- 导入失败文案统一
- 安装提示文案收敛

## 4. 推荐推进顺序

建议按以下顺序推进，不要再回到大规模结构改造：

1. 真机连续自用测试
2. 根据真机结果微调 segment 参数
3. 修阻塞阅读的问题
4. 冻结 `P1`
5. 进入 `P2` 的局域网 URL 书源

## 5. 当前风险提醒

- 当前 bundle 仍偏大，`pdf.js worker` 是主要体积来源
- segment 渲染策略已经具备，但仍需要真机验证参数是否合适
- 本地上传文件在刷新后不可直接继续阅读，这属于当前路线约束，不是偶发 bug

## 6. 当前建议

除阻塞阅读的问题外，`P1` 不再追加复杂书库管理功能。下一阶段默认进入 `P2` 规划与远程书源接入准备。
